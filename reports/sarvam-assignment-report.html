1.2 Architecture

The demo is a FastAPI server + single-page browser app. The browser captures mic audio via WebRTC, sends a turn at a time to FastAPI, which runs Saaras v3 STT → BM25-ish RAG over a one-page insurance brochure → Sarvam LLM with a voice-specific prompt → compliance post-filters → Bulbul v3 TTS, and returns audio plus telemetry to the browser.

    Browser (WebRTC mic + transcript + sources + metrics)
       │  audio blob               base64 WAV  ▲
       ▼                                       │
    FastAPI ── Orchestrator (state machine + metrics)
                  │
                  ├─▶ Sarvam STT (Saaras v3, auto-detect, mode=transcribe)
                  ├─▶ BrochureRAG (BM25-ish, 30 snippets, k=3)
                  ├─▶ Sarvam LLM (chat.completions, voice-style prompt)
                  ├─▶ Guardrails (claim-promise / premium-quote / tax / medical post-filters)
                  └─▶ Sarvam TTS (Bulbul v3, voice+pace from voice_design.py)

The production-migration exhibit is agents/livekit_agent.py — 80 lines, runs Aarav on LiveKit with Sarvam's plugin, with the four flags Sarvam's docs call out explicitly: flush_signal=True, turn_detection="stt", min_endpointing_delay=0.07, and no vad= on AgentSession.

2. Voicebot depth

2.1 Latency budget

Voice agents live or die at time-to-first-audio. Samvaad publishes <500 ms. The dev stack is honest about being slower because it's REST-first and runs from a laptop, but the budget below is what we're aiming for and reporting against:

StageTargetSarvam capability
End-of-speech detection50-100 msSarvam STT plugin emits start/end-of-speech events. min_endpointing_delay=0.07 in LiveKit.
STT (streaming Saaras v3)150-250 ms~70 ms processing latency. WebSocket streaming available since Jun 2025.
RAG retrieval50-150 msLocal BM25-ish over 30 snippets — sub-100 ms easy.
LLM first-token300-500 msStreaming chat.completions.
TTS first-audio (Bulbul v3)200-400 msWebSocket streaming since Jun 2025. 24 kHz default; 8/16/22/24/32/44/48 kHz supported.
Total time-to-first-audio~750-1300 msWithin ~2× of Samvaad's <500 ms benchmark on the dev stack — acceptable for a demo, and the report's honest about it.

The UI surfaces a stacked latency bar per turn so the interviewer can see exactly where time is spent.

2.2 Conversation design

The flow is a 7-state machine implemented in app/agent/orchestrator.py. Greeting → Discovery (3-5 short turns) → Pitch (one product, one reason, one cited feature) → Q&A (RAG-grounded, citations shown in UI) → Next-Steps / Handoff. Objection Repair is reachable from any state. Each state has an explicit purpose; transitions are decided per-turn from the user's intent. Crucially, the LLM does the language work — the state machine exists for metrics, pace control, and handoff capture, not for natural-language understanding.

2.3 The prompt

The single most consequential design decision is the prompt's first three lines:

"You are on a VOICE call — your text will be read aloud by a text-to-speech engine. Use short sentences. Avoid lists, markdown, headers, asterisks, code, parentheses, or numbers in digit form (write 'ten lakh', not '10,00,000'). Maximum 2-3 sentences per reply. Ask one question at a time. Never two."

Most "voicebot" prompts in the wild are reskinned chatbot prompts and produce TTS-hostile output (bullet lists with asterisks, parentheticals, ₹10,00,000 read as "rupees one zero comma zero zero comma zero zero zero"). Telling the LLM the output channel changes its behaviour materially.

2.4 Voice and pace selection

Bulbul v3 ships 37 voices across 11 languages. The default for Aarav is anand — warm, mid-pitch male, sounds 30s, not too corporate. The UI exposes a male/female toggle (Priya / Simran on the female side). Pace defaults to 1.0; the orchestrator drops to 0.9 when reading policy details (numbers, terms, exclusions) so users can follow. Voice identity is preserved across language switches — Anand answers in English, Hindi, or Hinglish without character change.

2.5 Multilingual / Hinglish

Saaras v3 with language="unknown" auto-detects, and Sarvam STT's codemix mode is what lets Aarav handle a sentence like "Bhai mujhe ek family floater chahiye monthly premium kitna hoga" without choking. Bulbul v3 handles English tokens inside a Hindi utterance natively, so the bot can answer in matching register.

2.6 Barge-in & turn-taking

The browser keeps the mic stream live while TTS is playing. A new user start-of-speech event pauses the <audio> element, cancels pending TTS generation, and treats the new utterance as the next turn. In the LiveKit production variant, this is handled by Sarvam STT's turn-detection signal — the configuration is one line: AgentSession(turn_detection="stt", min_endpointing_delay=0.07). No separate VAD model. (Sarvam's docs explicitly say: do not pass vad= to AgentSession.)

2.7 STT uncertainty repair

When Saaras returns confidence below ~0.7, Aarav repairs explicitly: "I caught 'family floater' but the second part wasn't clear — could you say it again?" This avoids the failure mode where the bot confidently answers the wrong question — the highest-cost error in insurance sales.

2.8 Compliance guardrails

Insurance is a regulated sale (IRDAI), and getting the compliance posture right is a CS-level judgment, not just an engineering one. Five bright lines, enforced as post-filter rewrites on the LLM reply:

Hard ruleWhat the bot says instead
Never quote a final premium"Honest answer — depends on age and a few health questions. For a family like yours we usually see fifteen to twenty-five thousand a year. The advisor will run the actual quote."
Never promise claim approval"Claims go through underwriting and waiting periods — the claims team makes the final call. I can flag your case for our advisor."
Never hide exclusions on direct askReads the relevant exclusion from the brochure.
No tax / legal / medical / investment advice"I can't advise on tax — your CA can confirm 80D and 80C, and our advisor can answer the policy-specific bit."
No final purchase without advisorAlways routes to handoff before close.

Every factual policy answer is grounded in a brochure snippet (shown as a "Source" card in the UI). The orchestrator tracks how often retrieval returns nothing useful — the rag_no_answer_rate metric is what a CSM would walk a Bajaj pilot through to identify gaps in the customer's KB.

5. Part 2 · Market research thesis

5.1 Thesis in one paragraph

India has, in 2026, made an explicit strategic bet on sovereign AI — not as protectionism, but as the only way to ensure a billion-plus Indian-language speakers get AI built for them rather than translated to them. The bet is being executed across four levers: subsidised compute (38,000+ GPUs deployed under the IndiaAI Mission at ₹150 / hour for selected startups), funded foundation models (Sarvam selected to build the sovereign LLM, alongside 11 other organisations), a DPI playbook that AI is being grafted onto (Aadhaar / UPI / DigiLocker / ONDC), and an unusually pragmatic policy stance — "regulate harms in deployment, not capability of models." Sarvam is the company positioned closest to the bullseye: Indic-first, voice-first, full-stack, already in production at Tata Capital, with an open-weight Apache 2.0 release of Sarvam-30B and Sarvam-105B in March 2026. The risks are real. The opportunity surface — a US$17 B Indian AI market by 2027 with the deepest concentration of Indic-language enterprise demand — is what makes Sarvam plausibly the most consequential Indian AI company of this cycle.

5.2 Strategic imperatives for sovereign AI in India

5.2.1 Language coverage is the moat global labs won't naturally close. GPT-class models trained on web data underperform materially on Indian languages — particularly low-resource scripts (Bodo, Santali, Maithili, Sindhi, Manipuri, Kashmiri) and on code-mixing patterns that dominate Indian speech (Hinglish, Tanglish, Banglish). India has 22 scheduled languages, 1,600+ dialects. A frontier lab in San Francisco optimising for English benchmarks has no commercial reason to fix this. An Indian lab does. Sarvam's Saaras v3 supports 22 languages; Bulbul v3 ships 11; Akshar reads 23.

5.2.2 Data sovereignty is becoming a real procurement criterion. The DPDP Act (in force, 2023) imposes data-residency obligations. Public-sector buyers, BFSI, healthcare, and large enterprises are RFP-mandating "data must not leave Indian jurisdiction." A US-hosted closed model can comply contractually but not architecturally. An India-hosted open-weight model can do both. Sarvam's "Sovereign by design — built and operated entirely in India" pitch on the Arya page is targeting exactly this procurement reality.

5.2.3 Cost-per-call is the gating factor for the 1B+ user opportunity. Frontier-model pricing translates to ~₹50-100 per voice-call-minute. That doesn't work for an Indian collections call placed because someone is ₹2,000 overdue. Sarvam-30B at ₹2.5 / million input tokens is roughly 40-100× cheaper than frontier. Saaras at ₹30 / hour of STT is comparable. This is the unit-economics moat that makes voice-AI viable in BFSI, citizen services, and edtech.

5.3 Policy landscape

IndiaAI Mission

Cabinet-approved March 2024 at a five-year outlay of ₹10,371.9 crore (~US$1.2 B). Seven pillars: compute capacity (₹4,563 cr — largest), foundation models (₹1,971 cr), startup financing (₹1,943 cr), AI Innovation Centres, datasets platform, applications, and safe & trusted AI. Run by IndiaAI (an Independent Business Division under Digital India Corporation, MeitY oversight).

Execution reality (April 2026): only ~₹400 cr disbursed in the first two years; ₹1,000 cr earmarked for FY27. 38,000+ GPUs deployed at the subsidised ₹150/hr rate. Sarvam, Gnani, Fractal, BharatGen and 8 others selected for foundation-model grants. The mission is making real moves but at slower pace than the headline outlay suggests — a watch-item for any vendor selling into the public AI ecosystem.

DPDP Act

In force. Consent, purpose limitation, data-fiduciary obligations. Cross-border transfers permitted but subject to a government negative list. Sensitive personal data (health, financial) effectively requires Indian residency in practice for risk-averse buyers. Samvaad's deployment matrix (Sarvam Cloud / VPC / on-prem / air-gapped) is the DPDP procurement story.

Adjacent policy

Digital India Bhashini (translation-mission corpus). MeitY's March 2024 AI advisory required "untested AI" deployment approval, walked back after industry pushback. Budget 2026 allocated ₹21,632.96 cr to MeitY overall, 17% down from FY26. India has explicitly not followed EU AI Act-style ex-ante regulation — the posture is "regulate harms in deployment, not capability of models," which is on net builder-friendly.

5.5 Sarvam's positioning

What Sarvam is, verified May 2026

Founded 2023, Bengaluru. ~US$54 M raised (Lightspeed, Peak XV, Khosla). Selected as the sovereign LLM builder under the IndiaAI Mission. Five products across three layers: foundation models (30B, 105B, Saaras v3, Bulbul v3, Vision, Translate, Mayura) · developer platform (dashboard.sarvam.ai, ₹1,000 free credits) · applications (Samvaad conversational agents, Studio content, Akshar doc digitisation, Arya enterprise agents, Indus). Open-source pivot in March 2026 — Sarvam-30B and Sarvam-105B released Apache 2.0 on Hugging Face and AIKosh. SOC 2 Type II + ISO certified.

Why Sarvam is positioned to win

Indic-first by design, not by translation. Models trained on Indic corpora and code-mixed data from day one. Bulbul v3's 37-voice library across 11 languages — orders of magnitude more native voice coverage than any global TTS.

Voice-first. ~80% of mass-market Indian users prefer voice over text for non-trivial interactions. Saaras+Bulbul is the only production-grade Indic voice stack at scale in 2026. Samvaad's <500 ms latency and 100M+ conversations are a real moat against any competitor that has to build voice from scratch.

Full-stack. Models → APIs → applications → enterprise platform. Most peers are layer-specific. Indian enterprise buyers (BFSI, govt) explicitly want one vendor; full-stack is winning enterprise deals.

Government tailwind. IndiaAI's chosen sovereign-LLM builder. Every public-sector AI RFP that lists "sovereign Indian model" as a requirement is, in effect, a sole-source for Sarvam.

Enterprise-grade from day one. SOC 2 Type II, ISO, DPDP-aligned, VPC + on-prem deployment. Unusual for a 2-year-old AI startup. Reflects who they sell to.

What Sarvam should worry about

Foundation-model commoditisation. If global open-weight models close the Indic gap via fine-tuning on Bhashini corpora — and they will, eventually — the model-layer moat compresses. Defense: push moat upward to voice (harder to commoditise) and applications (Samvaad, Arya).

Enterprise GTM execution. Indian BFSI sales cycles are slow, relationship-driven. Forward-deployed-engineer model is correct but expensive and people-bound. Scaling FDE while unit economics are still being worked out is the operational tension to watch.

The "I'll just use ChatGPT in Hindi" buyer. A meaningful share of SMBs won't pay for an Indic specialist if frontier-model quality on Indic narrows. Sarvam's answer: unit economics and on-prem availability — neither of which ChatGPT can match.

Hardware dependency. Like everyone, Sarvam runs on imported GPUs. Geopolitical squeeze would move the cost structure. Strategic, not near-term.

Forward-looking imperatives for Sarvam (my view)

  1. Productise Samvaad self-serve for mid-market. Today's contact-sales model captures enterprise but leaves mid-market (regional NBFCs, mid-insurers, edtech) on the table. A 14-day Samvaad self-serve tier would dramatically expand the funnel — and is what would have let me build Aarav inside Samvaad for this assignment.
  2. Telephony abstraction. Most buyers want a "Sarvam phone number" experience without integrating Twilio/Exotel. Sarvam-managed SIP would close more deals.
  3. Stronger Sarvam-LLM-in-LiveKit story. The official cookbook still uses OpenAI gpt-4o as the LLM in the LiveKit example. Replacing it with Sarvam-30B as the default is a high-leverage GTM move — every developer reading the docs ends up using a Sarvam LLM by default.
  4. Vertical depth in BFSI. "Samvaad for Collections" pre-packaged template with pre-built RAG over standard policy docs, escalation flows, and compliance guardrails would shorten the typical 4-6-week pilot to <2 weeks.
  5. Indic embeddings. A native sarvam-embed model would let Indian developers build Indic RAG without falling back to OpenAI / Cohere embeddings — closing the last layer where they reach for a US vendor.

5.7 So what — for a Sarvam CS hire

Customer Success at Sarvam in 2026 isn't onboarding-and-retention. It is:

  • Translating buyer outcomes into Sarvam's product surface — "you want a Hindi-Hinglish collections bot at <₹3 / call → that's Samvaad on Sarvam Cloud, here's the latency budget, here's the compliance guardrail config."
  • Being the early-warning system on what's missing — most enterprise customers in India are Sarvam's first deployment of a particular workflow. CS catches gaps before they become churn.
  • Owning "explain Sarvam's stack to the buyer's risk team" — DPDP, SOC2, deployment modes, model-update SLAs. Heavy on artefacts.
  • Carrying expansion — every deployed Samvaad customer is a candidate for Studio or Arya. CS is the natural seller.

The CS role is positioned at the highest-leverage point in Sarvam's GTM right now, because the product is mature enough to deploy but the playbook for who buys what when is still being written. That's what I'm applying for.
